## P4. ETL + LLM-обогащение

**Сюжет:** система автоматической обработки входящих инженерных задач с регулярным обновлением данных.

Этот этап превращает предыдущий сервис (LLM-Gate из P1–P3) в **данный пайплайн**, который регулярно загружает задачи из внешнего источника, обрабатывает их и сохраняет структурированные данные.

Главная цель: показать, что LLM может быть **частью data-pipeline**, а не только интерактивным сервисом.

---

# Теория (что нужно понимать)

## 1. ETL / ELT

**ETL**

```
Extract → Transform → Load
```

данные сначала трансформируются, потом сохраняются.

**ELT**

```
Extract → Load → Transform
```

данные сначала кладутся в хранилище, потом обрабатываются.

В современных системах чаще используют **ELT**.

Причина: дешевле хранить сырые данные и трансформировать позже.

---

## 2. Слои данных

Типичная архитектура:

```
Raw / Staging
↓
Cleaned
↓
Enriched
↓
Serving
```

### Raw

сырой источник (JSON/CSV/API)

### Clean

базовая нормализация

### Enriched

добавлены LLM-поля

### Serving

готовые данные для аналитики/продукта

---

## 3. DAG (Directed Acyclic Graph)

Airflow строит pipeline как граф задач.

Пример:

```
extract_issues
      ↓
normalize_data
      ↓
llm_enrichment
      ↓
quality_checks
      ↓
publish_table
```

Задачи:

* независимые
* повторяемые
* имеют retries

---

## 4. Идемпотентность

Pipeline должен быть **безопасен при повторном запуске**.

Пример:

плохой вариант

```
INSERT INTO table
```

хороший вариант

```
UPSERT / MERGE
```

---

## 5. Где LLM подходит в ETL

LLM хорош для:

* классификации
* извлечения сущностей
* нормализации текста
* категоризации

LLM плохо подходит для:

* агрегаций
* join
* deterministic transform

---

## 6. Data Quality Checks

После обработки должны быть проверки:

* null rate
* duplicate rate
* row count
* schema validation

# Практика: проект P4

## Сюжет

Ты строишь **AI-pipeline обработки задач разработки**.

Каждый день система:

1. Загружает новые задачи
2. Сохраняет raw данные
3. Нормализует структуру
4. Прогоняет через LLM-Gate
5. Проверяет качество
6. Сохраняет результат

# Задачи pipeline

## 1. Extract

Читает JSON:

```
data/sample_issues.json
```

пример:

```
{
"id": 123,
"title": "Checkout page returns 500",
"description": "...",
"author": "qa-team"
}
```

сохраняет в

```
raw_issues
```

---

## 2. Normalize

Приводит данные к нормальной схеме.

Пример:

* trim
* lower case
* remove html
* объединить title + description

результат

```
normalized_issues
```

---

## 3. LLM Enrichment

Для каждой записи:

```
POST /run/classify_v1
POST /run/extract_v1
```

результат:

```
label
entities
priority
summary
confidence
```

сохраняется в

```
enriched_issues
```

---

## 4. Quality Checks

Проверки:

### row_count

```
normalized == enriched
```

---

### null rate

```
label IS NOT NULL
```

---

### duplicates

```
unique issue id
```

---

Если проверка падает → DAG fail.
